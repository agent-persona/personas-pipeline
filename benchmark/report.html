Overview

The team produced 136 experiment branches across 6 researchers (Yash, Shruti, Max, Billy, Pak, Ivan). We built a deterministic 10-tenant synthetic benchmark (42 clusters, 3,305 records total), ran the pipeline on main as a baseline, then ran each branch that modifies runtime code and compared judge scores, groundedness, success rate, and cost. Backwards-compatible additions (new modules, new eval harnesses) were auto-merged without benchmarking. One branch (feat/adaptive-gower-segmentation) was evaluated separately in both Jaccard and Gower modes.

Methodology

The benchmark

10 deterministic synthetic tenants producing 42 clusters:

  • bench_mega_saas — 500 records, 6 cohorts (engineers, designers, PMs, marketers, sales, support)
  • bench_mega_fintech — 500 records, 5 cohorts (engineers, compliance, sales, security, data)
  • bench_mega_ecommerce — 500 records, 5 cohorts
  • bench_dense_devtools — 350 records, 4 cohorts
  • bench_sparse_30 — sparsity stress, 30 records
  • bench_sparse_60 — mid-sparsity, 60 records
  • bench_poisoned — 350 + 15 adversarial records (agriculture/culinary poison)
  • bench_heavy_tail — 300 records, 70/10/10/10 imbalance
  • bench_single_cohort — 200 records, 1 cohort — tests degeneracy
  • bench_diverse — 500 records, 8 cohorts

Classification

Each branch was bucketed by what files it modifies:

Runtime-changing (51 branches)
Modifies core pipeline files (prompt_builder, synthesizer, chat, judges, groundedness, segmentation). Default behavior differs from main, so a benchmark run is required to measure the delta.
Schema-additive (74 branches)
Adds new modules / persona variants without changing existing behavior. Auto-merge candidate.
Eval-only (11 branches)
Only adds measurement tooling under evals/. Auto-merge candidate.

Verdict rubric (for runtime-changing branches)

VerdictCriteria
MERGE ΔJudge ≥ +0.15 AND cost penalty < 30%, OR ΔGroundedness ≥ +0.05 AND ΔSuccess ≥ 0 AND cost penalty < 30%
REVIEW Mixed signals — positive judge but reliability concerns, or high variance across tenants
NEUTRAL All deltas within ±5% judge, ±2% groundedness, ±10% cost, ±5% success
REJECT ΔJudge < -0.15, OR ΔGroundedness < -0.05, OR cost penalty > 80%, OR ΔSuccess < -10%
REJECT-RELIABILITY ΔSuccess between -5% and -10% (functional but less reliable)
REJECT-BROKEN Zero personas produced — branch is fundamentally broken

Judge model

LLM-as-judge scoring 1-5 on five dimensions (grounded, distinctive, coherent, actionable, voice_fidelity) using Claude Haiku 4.5 with few-shot calibration (from exp-5.13). Every persona produced by every variant was scored — 1,867 + 42 + 39 + 21 = 1,969 total judge calls.

Scoreboard: 4-way comparison

Merged result: merging_experiments = main + 4 MERGE branches + 64 additive branches. After bumping MAX_RETRIES from 2 to 4 to absorb the stacked schema complexity, it ties main on reliability (42/42 personas) and beats it on judge score by +0.19 points.

The 4 MERGE-verdict branches

These are the high-impact branches that got full merges onto merging_experiments in descending-ΔJudge order. Smoke-tested after each merge; one conflict (exp-6.04 vs exp-2.07 in synthesizer.py parameter list) was resolved by keeping both additions.

Auto-merged branches (64)

Backwards-compatible additions — new files, no default-behavior changes. Auto-merged onto merging_experiments without benchmarking since their runtime impact is nil by construction.

REVIEW — mixed signals (7)

Judge score went up but reliability or cost hit warrants human review before merging.

NEUTRAL — within noise (19)

Within ±5% judge and ±5% success rate of main. Safe to merge as optional capability if the new parameter is useful, but no measurable lift on the default path.

REJECT-verdict branches (21)

Quality regression, reliability drop, or outright broken. Not merged.

Skipped / No-change branches (21)

Branches whose diff against main contained only filtered files (docs/plans/, output/, benchmark/, .gitignore) — nothing to bring in.
