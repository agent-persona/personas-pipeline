How to Use This

Not all dimensions
are created equal

The core insight: LLMs produce personas that are fluent but broken in specific, measurable ways — homogenized opinions, positivity bias, variance compression, stereotype amplification. These dimensions form a feedback loop: run evals → identify which dimensions fail → tune the generation pipeline → re-run. The eval is the thermometer; your prompts and data are the AC.

Key Learnings from This Run
Finding 01 · D23 Stereotype Amplification

Shared occupation was required

D23 needs ≥2 personas sharing an occupation to test demographic-trait correlation. Originally all 12 had unique jobs — D23 silently skipped. Dmitri was changed from "DevOps Engineer" to "Software Engineer" (same as Alice, opposite gender). D23 now runs and passes — no gender/occupation over-correlation at 85% threshold.

Finding 02 · D18 Joint Distribution — Score 0.106

The worst-scoring dimension in the run

Cramer's V detects stereotypical demographic correlations across the set. 0.106 means strong spurious correlations were found: income/occupation pairs, gender/lifestyle pairs. This is expected with only 12 personas — real-world datasets of 500+ would smooth these out. 12 is the minimum viable set, not the target.

Finding 03 · D29 Refusal Behavior — Score 0.000

Every persona answered every question

The scorer expects epistemic humility on out-of-scope questions — "I'm not sure about that" type responses. The mock data contained direct answers for every probe. This is a data fidelity issue, not a scorer bug. Real LLM-generated personas often over-answer; this dimension is a critical production check.

Finding 04 · D17 Calibration — Infrastructure fix

Scorer read from wrong source

CalibrationScorer only read confidence data from score_set(**kwargs). SuiteRunner never passes kwargs. Added fallback to read from source_context.extra_data["calibration_predictions"]. Now all 12 personas contribute calibration data — D17 runs and scores 0.700 (passes threshold at ECE < 0.15).

Finding 05 · D34 Multi-Turn Coherence — Data depth

20 turns minimum, not 6

The sliding-window algorithm needs window_size × 2 = 20 conversation turns minimum. Original mock data had 6 turns — scorer silently skipped. Expanded to 20 turns per persona. Result: 0.550 (below 0.6 threshold = fail), which is correct — the mock conversation drifts because it's synthetic.

Finding 06 · D19 Positivity Bias — Score 0.767 (FAIL)

RLHF makes every persona too sunny

Even with deliberate mock data including pain points and frustrations, D19 failed — the sentiment ratio of positive:negative markers exceeded the threshold. Bios emphasize achievements; responses trend toward optimism. This validates the scorer: LLMs systematically over-produce positive-valence content, and this dataset is no exception.

Vocabulary register mismatch — personas with "basic" or "intermediate" vocabulary_level still produce doctoral-register prose. LLMs cannot write below their own training distribution. Detected via embedding similarity to basic vs. advanced register prototypes.

Hedge-Word Inflation

Systematic overuse of meta-commentary hedge phrases regardless of persona tone. A "blunt" persona should never say "I'd be happy to elaborate" — but LLMs inject these phrases uniformly. Detected via static phrase list (~30 phrases) normalized per 100 words, thresholded by communication tone.

0.000

0.000

What to Do With This

Next steps

Immediate

Connect to the Generation Pipeline

Wire the eval CLI to real LLM-generated persona output. The golden dataset validates the framework; now it needs to measure actual production personas.

  • Run evals run-set on first real persona batch
  • Identify which dimensions fail on real data vs. golden
  • Establish baseline scores per dimension
Immediate

Define Pass Thresholds for Your Use Case

Current thresholds are research defaults. Your application (twin chat, synthetic surveys, UX research) may need different tolerances — especially for bias dims.

  • Set minimum thresholds for D6, D13, D19 (homogenization)
  • Configure D40 cost/latency limits for your budget
  • Decide whether D22 (hyper-accuracy) is a concern in your domain
Short-Term

Use Failures to Tune Prompts

Each failing dimension points directly at a specific prompt or sampling fix. Don't guess — the eval tells you exactly what broke.

  • D19 failing → add adversity/challenge prompts
  • D13 failing → increase temperature or add diversity constraints
  • D6 failing → deduplicate against existing persona embeddings
  • D18 failing → sample from real demographic joint distributions
Short-Term

Wire into CI/CD

The GitHub Actions workflow is already scaffolded. Connect it to run on every persona generation PR. Tier 1 failures should block merges; Tier 2+ should alert but not block.

  • Fast tests (no LLM, no embedding): <30s
  • Slow tests (embedding): ~2 minutes
  • Set Slack webhook for Tier 1 failures
Medium-Term

Build a Scoring Dashboard

Store results in Postgres, track scores over time. D41 (Degradation Detection) and D37 (Temporal Stability) become meaningful only when you have historical data to compare against.

  • Score each dimension over time
  • Alert on >1σ drift from baseline
  • Track per-model, per-prompt-version breakdowns
Medium-Term

Replace Golden Dataset with Real Personas

The 12-persona golden dataset validates the framework works. Once you have 50–100 real generated personas, replace the golden set — real data makes the eval more meaningful for your specific domain.

  • Keep golden dataset as a smoke test baseline
  • Build domain-specific source text corpus
  • Curate tail-insight benchmark from real outlier users
